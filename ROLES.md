# 角色定义（ROLES）

> **单一信息源**：本项目所有角色定义以此文档为准。PROJECT_CONTEXT、WORKFLOW、规则文件等引用本文档，不重复定义。

---

## 协作顺序

```
产品经理 → UI/UX 设计师 → 架构师 → Tech Lead → 开发团队 → 测试团队
     ↑
项目经理贯穿全程
```

---

## 七方角色

| 角色 | 职责 | 产出物 | 依赖 |
|------|------|--------|------|
| **项目经理** | 协调流程、分配任务、跟踪状态、维护文档、为各角色生成 task prompt | `WORKFLOW.md`、BACKLOG 状态、各 agent 的 task prompt | 无 |
| **产品经理** | 定义需求、范围、输出格式；拍板决策；**不负责 UI 设计稿** | `MVP1_PRD.md`、`INPUT_PARAMS_SPEC.md`、`OUTPUT_FORMAT_SPEC.md`、`UI_REDESIGN_SPEC.md`（需求文档） | 无 |
| **UI/UX 设计师** | 界面设计、交互流程、视觉规范、可预览设计稿 | 设计说明文档；`frontend/preview/` 或约定路径下的静态 HTML 预览页 | 产品经理的 UI 需求文档 |
| **架构师** | 设计架构、目录结构、演进路径 | `ARCHITECTURE.md`、BACKLOG 初版 | 产品经理产出 |
| **Tech Lead** | 拆解任务、制定子任务与验收标准 | `BACKLOG.md` 中的任务分解 | 架构师 + 产品经理 + UI/UX 设计稿（若有 UI 任务） |
| **开发团队** | 按 BACKLOG 实现代码 | `src/`、`backend/`、`frontend/` 代码 | Tech Lead 拆解后的任务 |
| **测试团队** | 验证功能、输出、边界；提出测试用例 | 测试建议、验收反馈 | 开发实现 + 规范文档 |

---

## 各角色详细说明

### 项目经理

- **职责**：整体把控、管理进度文档、确定下一任务、为 responsible 角色生成 prompt、接收汇报并更新状态。
- **产出**：`WORKFLOW.md`、BACKLOG 状态、各 agent 的 task prompt。
- **不做**：定义需求（产品）、设计架构（架构师）、拆任务（Tech Lead）、写代码（开发）、执行测试（测试）。
- **依赖**：无。

### 产品经理

- **职责**：需求定义、用户故事、功能优先级；定义输入输出规范；拍板决策。
- **产出**：`MVP1_PRD.md`、`INPUT_PARAMS_SPEC.md`、`OUTPUT_FORMAT_SPEC.md`；UI 改版时产出 `UI_REDESIGN_SPEC.md`（需求与验收标准）。
- **不做**：不负责可预览的 UI 设计稿，该工作由 UI/UX 设计师负责。
- **依赖**：无。

### UI/UX 设计师

- **职责**：界面设计、交互流程、视觉规范、原型；产出可预览的设计稿供确认。
- **产出**：设计说明文档；`frontend/preview/` 或项目约定路径下的静态 HTML 预览页。
- **触发**：产品经理产出 UI 需求文档（如 `UI_REDESIGN_SPEC.md`）后，开发实施前。
- **协作**：设计稿经确认后，Tech Lead 再拆解任务；若有变更，同步更新需求文档。
- **依赖**：产品经理的 UI 需求文档。

### 架构师

- **职责**：系统设计、技术选型、模块划分、演进路径。
- **产出**：`ARCHITECTURE.md`、`BACKLOG.md` 初版。
- **依赖**：产品经理产出。

### Tech Lead

- **职责**：技术落地、代码规范、技术难点；拆解任务、制定子任务与验收标准。
- **产出**：`BACKLOG.md` 中的子任务（如 1.1、1.2…）及验收标准。
- **依赖**：架构师、产品经理；若有 UI 任务则依赖 UI/UX 设计稿。

### 开发团队

- **职责**：编码实现、功能开发。
- **产出**：`src/`、`backend/`、`frontend/` 代码；BACKLOG 状态更新。
- **依赖**：Tech Lead 拆解后的任务。

### 测试团队

- **职责**：测试用例、质量验证、回归测试；对照规范验证实现。
- **产出**：验收结论、问题列表、测试建议。
- **依赖**：开发实现、规范文档。
