# 端形态决策

> 本文档记录 MVP1 上线端形态的决策结论及理由。
>
> 决策日期：2026-02-22

---

## 1. 决策结论

**选定端形态：Web**

---

## 2. 决策理由

| 维度 | 分析 |
|------|------|
| **用户触达** | 目标用户为中国用户，规划新西兰自驾行程。用户多在 PC 或手机上做行程规划，以浏览器或微信内打开为主。Web 可同时覆盖两端，响应式设计即可满足移动端体验。 |
| **开发成本** | 后端已为 Python，接入 Web 前端（如 FastAPI + 简单前端 / Streamlit）成本最低。表单输入 + 结果展示的交互简单，无需原生能力。 |
| **上线速度** | 部署到服务器即可上线，无需应用商店或微信审核，迭代周期最短，符合 MVP 快速验证目标。 |
| **分发与推广** | 通过链接即可分享，方便内测、朋友试用、社群推广；可配合短链、二维码做传播。 |

---

## 3. 补充考量（复盘讨论）

### 3.1 中国用户与 Web

中国用户整体更习惯小程序、App，单独使用浏览器访问 Web 的比例较低。但：
- **获客渠道**：早期用户可能来自小红书。小红书以链接分享为主，用户点击后可在浏览器或应用内打开，Web 链接可无缝承接。
- **当前资质**：暂无企业主体和微信小程序资质，冷启动阶段选择 Web 可避免资质与审核门槛。

### 3.2 未来市场方向

产品规划中，未来可能面向**国际市场**，为境外游客入境中国游做行程规划。Web 可同时服务于：
- 国内：通过小红书等渠道链接获客
- 国际：境外用户习惯使用浏览器、搜索引擎，Web 为通用形态

若优先做小程序，海外无法使用，后续需另建 Web/App，开发成本翻倍；Web 先发可覆盖两个市场。

### 3.3 App 的取舍

| 形态 | 优势 | 为何不在 MVP1 选择 |
|------|------|--------------------|
| **微信小程序** | 触达中国用户便捷，可在微信内分享、无需下载 | 需企业主体、备案域名、微信审核；暂无资质；海外无法使用。 |
| **App（原生）** | 体验好，可做推送、离线；国内外均可上架 | 需下载安装，冷启动门槛高；MVP 验证阶段，点击即用优于「先下载再体验」；开发与维护成本大。 |

---

## 4. 与另外两种形态的取舍（简要）

| 形态 | 为何不在 MVP1 选择 |
|------|--------------------|
| **微信小程序** | 需企业主体、备案、微信审核，暂无资质；海外市场无法覆盖。 |
| **App** | 下载安装摩擦大；需双端开发；对 MVP 单轮表单 + 行程展示而言投入过高。 |

---

## 5. 后续规划

- **MVP1**：以 Web 为主，建议响应式设计，兼顾手机访问；获客可通过小红书链接分享。
- **验证通过后**：可评估微信小程序作为中国区第二端；App 可视业务量再决策。
